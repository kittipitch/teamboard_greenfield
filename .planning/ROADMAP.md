# Roadmap: TeamBoard

## Overview

Three phases build the complete app from the ground up: a safe storage layer first, then a curl-testable REST API, then the browser UI that delivers every user-facing requirement. Each phase is independently verifiable before the next depends on it.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 1: Data Foundation** - Atomic JSON storage, card schema, crash-safe writes
- [ ] **Phase 2: REST API** - Four CRUD endpoints with input validation and XSS prevention
- [ ] **Phase 3: Board UI** - Three-column board with full add/edit/delete card interactions

## Phase Details

### Phase 1: Data Foundation
**Goal**: A safe, crash-resistant storage layer exists that all other code can depend on
**Depends on**: Nothing (first phase)
**Requirements**: CARD-04
**Success Criteria** (what must be TRUE):
  1. Cards written to cards.json survive a process restart and reload correctly on startup
  2. Two simultaneous writes do not corrupt cards.json (atomic, serialized writes via lowdb/steno)
  3. A truncated or malformed cards.json on startup does not crash the server — it falls back to an empty board
  4. Every card has a unique ID generated by crypto.randomUUID() (no Date.now() or Math.random())
**Plans**: 2 plans

### Phase 2: REST API
**Goal**: All four card operations are accessible via HTTP and verifiable with curl before any browser UI exists
**Depends on**: Phase 1
**Requirements**: (infrastructure — no discrete v1 requirement; enables CARD-01, CARD-02, CARD-03 in Phase 3)
**Success Criteria** (what must be TRUE):
  1. `curl GET /api/cards` returns the current card list as JSON
  2. `curl POST /api/cards` with valid body creates a card and returns it; text over 500 chars is rejected with 400
  3. `curl PUT /api/cards/:id` updates card text; `curl DELETE /api/cards/:id` removes the card
  4. Card text is HTML-escaped before being written to storage (XSS prevention)
  5. An unknown column slug in POST is rejected with 400; a missing card ID in PUT/DELETE returns 404

### Phase 3: Board UI
**Goal**: Users can open the board in a browser and add, edit, and delete cards across all three columns with no login
**Depends on**: Phase 2
**Requirements**: BOARD-01, BOARD-02, BOARD-03, CARD-01, CARD-02, CARD-03
**Success Criteria** (what must be TRUE):
  1. User sees three labeled columns — Went Well, To Improve, Action Item — on a single page without logging in
  2. User can type text into a form in any column and submit it; the new card appears in that column immediately
  3. User can click to edit an existing card's text inline and save the change without a full page reload
  4. User can delete any card and it is removed from the column immediately
  5. Each column header shows the current card count; an empty column shows helpful guidance text instead of a blank space
